概念

Cache Probing正如其名就是缓存探测，也是Leak众多方法中的一种。

某些资源会被浏览器缓存起来

某些情景下，如果攻击者能够通过某种信道探测到这个资源加载的状态，这个状态的信息可以告诉攻击者，受害者是否之前访问过这个资源

攻击手法

Timing

也是最容易想到的，自然而然从浏览器本地缓存中加载和网上加载的所用的时间不同，那么就依据返回的时间来判断加载状态

最好用的办法其实就是await fetch(url, { 'cache': 'force-cache' })

然后获取时间就行了

Error Events

CORS

考虑一下这种情况

如果从target.com请求托管在server.com上的资源，则在响应头会返回：Access-Control-Allow-origin:target.com。如果资源已缓存，则此信息将与资源一起存储在浏览器缓存中。因此，如果攻击者attacker.com试图获取相同的资源，则有两种可能的情况：

  • 该资源不在缓存中：该资源可以与Access-Control-Allow-Origin:attacker.com标头一起提取和存储。

  • 资源已在缓存中：尝试从缓存中提取资源，但由于ACAO标头值与请求来源不匹配，它也将生成CORS错误。

限制

从别人博客偷的图

可以很详细的看到缓存的资源也是存在隔离的，cache key 只看 eTLD(有效顶级域名,例如.com.cn,.github.io等)+1

以上图为例第一次请求，那么Cache Key:{xx}就包含了窗口的网址，内嵌窗口的网址，资源的地址

如果攻击者想要访问到这次的缓存，那么他的Cache Key:{attacker-site,attacker-site,http://x.example/doge.png}，自然而然不能访问到该图片的缓存，因为这里有两个值都不相同

而且同源策略似乎还是存在的，也就是同源才能分享缓存

不过我看大哥说的是在某些版本的headless chrome是不会管Cache Key的，所以出现了这样的配置，就不用考虑这么多了，而且子域名和端口都是被忽略掉的，比如http://wqd.example.org最后只会存http://example.org下来
